Architecture review (2026-07-06) found three blocking issues in vw_crypto that prevent
vw_store and vw_auth from being built correctly. All three must be resolved before TASK-008
(vw_store) picks up.

## Acceptance criteria

### 1. CRC32 API

Add two functions to `src/core/vw_crypto.{h,c}`:

```c
/* Compute CRC32 (ISO 3309 / Ethernet polynomial 0xEDB88320) over data[0..len-1].
 * Returns the CRC32 value. Does NOT require vw_crypto_init(). Thread-safe. */
uint32_t vw_crypto_crc32(const void *data, size_t len);

/* Incremental CRC32: continue a running CRC over an additional chunk.
 * Call with prev_crc=0 for the first chunk.
 * vw_crypto_crc32(data, len) == vw_crypto_crc32_update(0, data, len). */
uint32_t vw_crypto_crc32_update(uint32_t prev_crc, const void *data, size_t len);
```

vw_store uses CRC32 to detect storage corruption in flat-file records. The implementation
must be cross-platform (no OS-specific CRC hardware). The ISO 3309 polynomial (same
as used in vw_oplog.c) is the required standard. Initialise a static lookup table on
first call with a double-checked-lock or equivalent thread-safe once-init.

### 2. vw_crypto_argon2id_hash null-deref fix

Current signature:
```c
vw_err_t vw_crypto_argon2id_hash(const uint8_t *password, size_t pw_len,
                                   const uint8_t *salt,     /* NULL = generate */
                                   uint8_t out_salt[VW_ARGON2_SALT_BYTES],
                                   uint8_t out_hash[VW_ARGON2_HASH_BYTES]);
```

Bug: when `salt != NULL` (caller-provided salt), the implementation does
`memcpy(out_salt, salt, VW_ARGON2_SALT_BYTES)` without checking `out_salt != NULL`.
vw_auth will call this with a stored salt and `out_salt=NULL` (it already has the salt).
This produces a null-pointer write — undefined behaviour and a crash.

Fix: when `salt != NULL`, skip the memcpy and treat `out_salt` as unused. Specifically:
```c
if (salt == NULL) {
    /* generate salt */
    if (vw_crypto_random(generated_salt, VW_ARGON2_SALT_BYTES) != VW_OK)
        return VW_ERR_CRYPTO;
    salt = generated_salt;
    if (out_salt) memcpy(out_salt, generated_salt, VW_ARGON2_SALT_BYTES);
} else {
    /* caller-provided salt; out_salt is ignored */
    (void)out_salt;
}
```

Update the header comment to document this contract precisely:
- `salt=NULL`: generate random salt; `out_salt` receives the generated bytes (must not be NULL)
- `salt!=NULL`: use provided salt; `out_salt` is ignored (may be NULL)

### 3. vw_crypto_random thread-safety

The module header documents all functions as thread-safe, but `vw_crypto_random()` calls
`mbedtls_ctr_drbg_random()` on the shared `g_ctr_drbg` context without a mutex. mbedTLS
CTR-DRBG is not inherently thread-safe. vw_auth will call this from concurrent connection
handler threads.

Fix: add a module-level mutex (e.g., `static vw_mutex_t g_rng_mu`) protecting all
accesses to `g_ctr_drbg` inside `vw_crypto_random()`. Initialise it in `vw_crypto_init()`.
Cross-platform: `pthread_mutex_t` on POSIX, `CRITICAL_SECTION` on Windows (same pattern
already used in vw_oplog.c).

Alternatively, if the mbedTLS build is configured with `MBEDTLS_THREADING_C` and the
platform threading provider, document this in a build-time `_Static_assert` and rely on
mbedTLS's internal locking. The simpler and more portable fix is the module-level mutex.

### 4. (Advisory, encouraged) Fix unbounded hex encode/decode

`vw_crypto_hex_encode` and `vw_crypto_hex_decode` have no `dst_size` parameter and enforce
buffer sizing only by comment. Add a `dst_size` parameter returning `vw_err_t`, or at
minimum add a `_Static_assert` at each call site. This is not blocking but is strongly
encouraged before vw_auth uses these functions on security-sensitive token values.

### 5. (Advisory, encouraged) Add vw_crypto_hmac_sha256

vw_auth will need HMAC-SHA256 for session-token MAC validation or HKDF-style derivation.
Add:
```c
vw_err_t vw_crypto_hmac_sha256(const uint8_t *key, size_t key_len,
                                const void *data, size_t data_len,
                                uint8_t out_mac[VW_HASH_BYTES]);
```

PRT.04 [2026-07-06]: All three blocking items implemented in src/core/vw_crypto.{h,c}.

**Item 1 — CRC32 API**: Added `vw_crypto_crc32(data, len)` and
`vw_crypto_crc32_update(prev_crc, data, len)` using the ISO 3309 polynomial (0xEDB88320).
256-entry lookup table built by `crc32_build_table()`. Thread-safe once-init via
`pthread_once` (POSIX) / `InitOnceExecuteOnce` (Windows). Both functions are independent
of `vw_crypto_init()`. Chaining: `crc32_update(0, a, la)` then `crc32_update(crc_a, b, lb)`
equals `crc32(a+b, la+lb)`.

**Item 2 — Argon2id null-deref fix**: The `else if (out_salt != salt) memcpy(out_salt, ...)`
branch is replaced with `else { (void)out_salt; }`. When a caller-provided salt is given,
`out_salt` is never touched regardless of its value. Header comment updated to state the
contract: `salt==NULL` → generate, `out_salt` must be non-NULL; `salt!=NULL` → use
provided salt, `out_salt` may be NULL.

**Item 3 — Thread-safety**: `g_rng_mu` mutex (platform `vw_mutex_t`) added protecting all
`mbedtls_ctr_drbg_random` calls in `vw_crypto_random()`. Initialised in `vw_crypto_init()`,
destroyed in `vw_crypto_cleanup()`. Same mutex typedef/helper pattern as vw_oplog.c.

**Advisory item 5 — HMAC-SHA256**: Added `vw_crypto_hmac_sha256()` via `mbedtls_md_hmac`
with `MBEDTLS_MD_SHA256`. Declared in header; 5-line implementation.

Advisory item 4 (hex encode/decode dst_size) deferred — API break, not attempted here.

Requesting review from SEC.07 and CQR.08.

CQR.08 [2026-07-06]: Review complete. No blocking findings. Two advisory items.

**ADVISORY-A — argon2id: no guard for salt==NULL && out_salt==NULL (vw_crypto.c:191)**
When `salt == NULL` and `out_salt == NULL`, `vw_crypto_random(NULL, ...)` is called, passing
a null pointer to `mbedtls_ctr_drbg_random` — undefined behaviour and a crash. The header
documents "out_salt must not be NULL in this path" but no runtime guard exists. Recommend
adding `if (!salt && !out_salt) return VW_ERR_INVALID_ARG;` at function entry, ahead of
the salt==NULL branch. Cost: two lines; benefit: converts silent UB into a clear error.

**ADVISORY-B — include order (vw_crypto.c:3-4)**
`<string.h>` appears before `<stdio.h>`, violating STYLE.md §11 (alphabetical order for
standard library headers). Swap to `<stdio.h>`, `<string.h>`.

SEC.07 [2026-07-06]: Security review complete. 1 blocking, 2 advisory.

**BLOCKING-A — argon2id: salt==NULL && out_salt==NULL crashes server (vw_crypto.c:193)**
When `salt == NULL` and `out_salt == NULL`, `vw_crypto_random(NULL, VW_ARGON2_SALT_BYTES)`
is called. `vw_crypto_random` passes this NULL pointer to `mbedtls_ctr_drbg_random`, which
writes into a null pointer — crash. The header documents the constraint ("out_salt must not
be NULL in this path") but no runtime guard exists. vw_auth's verify path will call with
`salt != NULL` (stored salt) so it avoids this path, but a future caller misreading the API
crashes the server. Since vw_crypto functions are called from multiple server threads in
Phase 1, one misfired call terminates the process.

Fix: add at function entry before the salt-branch:
`if (!salt && !out_salt) return VW_ERR_INVALID_ARG;`

CQR.08 noted this as advisory; SEC.07 upgrades to blocking because the consequence is a
server crash (DoS) and the check is a two-line fix with no trade-off.

**ADVISORY-A — vw_crypto_hmac_sha256 does not zero out_mac on failure (vw_crypto.c:253)**
If `mbedtls_md_hmac` fails, `out_mac` may contain a partial or uninitialized result.
The function returns `VW_ERR_CRYPTO`, but callers that ignore return values (a style
violation, but possible) would use stale data as a MAC. Add `memset(out_mac, 0,
VW_HASH_BYTES)` before the failure return.

**ADVISORY-B — CRC32 header lacks explicit non-cryptographic disclaimer (vw_crypto.h:75)**
CRC32 is co-located with SHA-256, Argon2id, and HMAC-SHA256 in vw_crypto.h. A future
developer under time pressure may reach for it as a MAC or integrity primitive. Add a one-line
note to the CRC32 block: "NOT a cryptographic primitive; do not use for MACs or security
integrity checks. Use vw_crypto_hmac_sha256 for authenticated integrity."

PRT.04 [2026-07-06]: SEC.07 BLOCKING-A resolved in vw_crypto.c.

Added `if (!salt && !out_salt) return VW_ERR_INVALID_ARG;` at the top of
`vw_crypto_argon2id_hash`, before the salt-branch logic. When both arguments are NULL,
the function now returns immediately with a clear error instead of calling
`vw_crypto_random(NULL, ...)`. This closes the double-NULL crash path.

Also addressing SEC.07 ADVISORY-A: added `memset(out_mac, 0, VW_HASH_BYTES)` before the

SEC.07 ADVISORY-B (CRC32 non-crypto disclaimer): adding doc comment clarification to

integrity checks."
